# Опис архітектури Kubernetes для Restaurant Booking System

## 1. Сервіс
Ми моделюємо сервіс **Restaurant Booking API**. Він відповідає за управління столиками та бронюваннями.

## 2. Розміщення в кластері
Сервіс буде розміщено у **namespace: restaurant-prod**. Це дозволяє ізолювати продуктивне середовище від тестового.

## 3. Шлях трафіку (Ingress)
1.  Користувач заходить на домен `api.restaurant-booking.com`.
2.  Запит потрапляє на **Ingress Controller**.
3.  Ingress перевіряє шлях (наприклад `/reservations`) і перенаправляє запит на внутрішній **Service** (`restaurant-svc`).
4.  Service балансує навантаження між доступними подами.

## 4. Масштабування
Ми заклали **2 репліки (Pods)** для базової відмовостійкості. Якщо один под впаде, інший продовжить обробляти запити без простою системи.

## 5. CI/CD Pipeline
1.  Розробник робить `git push` у GitHub.
2.  GitHub Actions запускає тести та збирає Docker-образ.
3.  Образ пушиться в **GitHub Container Registry (GHCR)**.
4.  Deployment у Kubernetes оновлюється, завантажуючи нову версію образу з Registry.
